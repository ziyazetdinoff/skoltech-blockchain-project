# Техническое задание

## Проект

**Название:** DCA Investing Smart Contract (Interval-Based Buying) with Automation  
**Формат:** учебный проект, MVP  
**Язык документа:** русский  
**Целевая сеть:** Ethereum Sepolia  
**DEX:** Uniswap  
**Интерфейс:** Telegram-бот  
**Результат:** код, деплой в testnet, простой интерфейс, отчет

---

## 1. Цель проекта

Разработать минимально рабочую систему для стратегии Dollar-Cost Averaging (DCA), в которой пользователь может создать инвестиционный план и автоматически покупать один токен за другой через фиксированные интервалы времени.

Пример сценария: пользователь создает план на покупку WETH за USDC на сумму 100 USDC каждую неделю. Средства резервируются в смарт-контракте, а off-chain executor периодически проверяет, какие планы пора исполнять, и вызывает выполнение swap через Uniswap с защитой по slippage.

- Пополнение плана и вывод неиспользованного остатка.
- Автоматическое исполнение плана через off-chain executor.
- Интеграция с Uniswap для выполнения swap.
- Защита от проскальзывания через `minAmountOut`.
- Развертывание в Ethereum Sepolia.

- Простой Telegram-бот для управления проектом.
- Отчет по проекту.

### 2.2 Что не входит в MVP

- Поддержка нескольких DEX.
- Поддержка сложной маршрутизации обменов.
- Полноценный веб-интерфейс.
- Многофакторная авторизация и production-grade хранение секретов.

- Стимулирование внешних исполнителей через reward-механизм.
- Полноценная multi-user SaaS-архитектура.
- Chainlink Automation в основной реализации.

### 3.1 Выбранный стек

- Смарт-контракты: `Solidity`
- Среда разработки и деплоя: `Hardhat`

### 3.2 Обоснование решений

## 4. Ключевые допущения

- Проект реализуется как **single-user demo MVP**.
- Telegram-бот и backend работают для одного тестового пользователя/кошелька, заданного в конфигурации.
- Для простоты в MVP используются ERC-20 токены. Для покупки ETH в демонстрации используется `WETH`.
- Первая версия поддерживает swap через один пул Uniswap (`single-hop`).
- Первая версия может быть ограничена одной преднастроенной парой, например `USDC -> WETH`, но код контракта желательно писать так, чтобы он не был жестко привязан к одной паре.
- Газ за исполнение платит off-chain executor.

## 5. Бизнес-логика MVP

### 5.1 Модель работы DCA-плана

Пользователь заранее резервирует бюджет плана в смарт-контракте. Контракт хранит средства пользователя и исполняет по одной покупке за один интервал.

Каждый план содержит:

- токен списания (`tokenIn`)
- токен покупки (`tokenOut`)
- сумму одной покупки (`amountPerInterval`)
- размер интервала в секундах
- общий бюджет плана (`totalBudget`)
- остаток бюджета (`remainingBudget`)
- максимально допустимое проскальзывание (`slippageBps`)
- адрес получателя купленного актива
- время начала (`startTime`)
- время следующего исполнения (`nextExecutionTime`)
- статус плана

### 5.2 Модель финансирования

Для MVP выбирается модель **предварительного депозита всего бюджета**:

- пользователь одобряет контракту списание `tokenIn`
- при создании плана контракт переводит `totalBudget` на свой баланс
- при каждом исполнении расходуется `amountPerInterval`
- пользователь может пополнить план отдельной транзакцией
- пользователь может вывести неиспользованный остаток после отмены или частично, если это разрешено логикой контракта

### 5.3 Поведение при пропущенных интервалах

Если executor не работал и один или несколько интервалов были пропущены, старые интервалы не догоняются. При следующем доступном запуске выполняется только одна покупка за текущий актуальный интервал.

### 5.4 Поведение при неуспешном swap

Если swap не исполнился из-за:

- высокого slippage
- нехватки ликвидности
- временной ошибки сети

транзакция исполнения откатывается, а план остается активным. Executor должен попробовать выполнить его позже повторно.

## 8. Архитектура системы

### 8.1 Компоненты

Система состоит из четырех основных компонентов:

1. `DCAPlanManager`  
   Смарт-контракт, который хранит планы, средства и логику исполнения.

2. `Uniswap Router`  
   Внешний контракт DEX, через который выполняется swap.

3. `Executor Service`  
   Node.js процесс, который по таймеру ищет планы, готовые к исполнению, и отправляет транзакции.

4. `Telegram Bot`  
   Интерфейс для управления планами и просмотра статусов.

### 8.2 Общая схема взаимодействия

1. Пользователь через Telegram-бота инициирует создание плана.
2. Backend формирует и отправляет транзакцию в контракт.
3. Контракт резервирует бюджет и сохраняет план.
4. Executor по расписанию проверяет активные планы.
5. Для просроченного плана executor получает котировку и рассчитывает `minAmountOut`.
6. Executor вызывает `executePlan`.
7. Контракт делает swap через Uniswap.
8. Полученный `tokenOut` отправляется получателю.
9. Contract обновляет `remainingBudget` и `nextExecutionTime`.
10. Backend показывает обновленный статус через Telegram-бота.

## 9. Модель данных смарт-контракта

### 9.1 Сущность плана

Рекомендуемая структура `Plan`:

```solidity

    address owner;
    address recipient;
    address tokenIn;
    address tokenOut;
    uint256 amountPerInterval;
    uint256 totalBudget;
    uint256 remainingBudget;
    uint256 intervalSeconds;
    uint256 slippageBps;
    uint256 startTime;
    uint256 nextExecutionTime;
    bool active;
    bool paused;
    bool canceled;

```

Допускается скорректировать структуру при реализации, если логика станет проще и чище.

### 9.2 Идентификация

- Каждый план получает уникальный `planId`.
- Контракт хранит счетчик `nextPlanId`.
- Доступ к плану осуществляется по `planId`.

### 9.3 Статусы плана

План может находиться в одном из логических состояний:

- `Active`
- `Paused`
- `Canceled`
- `Completed`

`Completed` можно определять вычисляемо, например когда `remainingBudget < amountPerInterval`.

## 12. Логика исполнения плана

### 12.1 Условия готовности к исполнению

План считается готовым к исполнению, если одновременно выполняются условия:

- план активен
- план не на паузе
- план не отменен
- `block.timestamp >= nextExecutionTime`
- `remainingBudget >= amountPerInterval`

### 12.2 Расчет следующего времени исполнения

После успешного исполнения:

- `remainingBudget` уменьшается на `amountPerInterval`
- `nextExecutionTime` сдвигается вперед на один интервал относительно текущего времени так, чтобы старые интервалы не догонялись

Рекомендуемое поведение:

- если executor пришел поздно, выполняется только одна покупка
- после исполнения `nextExecutionTime = block.timestamp + intervalSeconds`

### 12.3 Расчет slippage protection

`minAmountOut` не рассчитывается полностью внутри контракта. Для MVP он рассчитывается off-chain:

1. executor получает ожидаемый `quoteAmountOut`
2. executor применяет `slippageBps`
3. executor передает `minAmountOut` в `executePlan`
4. Uniswap гарантирует, что swap пройдет только если результат не меньше `minAmountOut`

## 22. Итоговая формулировка MVP

Нужно реализовать учебную систему DCA на Ethereum Sepolia, состоящую из смарт-контракта `DCAPlanManager`, Node.js executor-сервиса и Telegram-бота. Пользователь должен иметь возможность зарезервировать бюджет в контракте, настроить интервальный план покупки токена через Uniswap, а backend должен автоматически выполнять покупки по расписанию с учетом slippage protection. Решение должно быть простым, демонстрационным, покрытым unit tests и сопровождаться коротким отчетом.
